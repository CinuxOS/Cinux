## 发现

### DEBT-002 精确坐实（升级证据）— D4 I1/I2/I3 fail

production Task 退出链断裂：`sys_exit`→Zombie→`waitpid` reap→Dead / `exit_current`→Dead，**全程无 `delete Task`、无 `release_resources`**。`remove_task`（scheduler.cpp:190）是唯一 delete 入口，但**仅 test/ 调用，production 死代码**。后果：每个退出进程泄漏 Task 结构（~1KB）+ sig_actions/fd_table/cwd refcount 引用（不递减→共享对象连锁泄漏）+ 核栈（16KB）+ addr_space（页表+VMA）。P1 长期泄漏（shell/fork-exec 高频场景累积）。Q4 头号修复候选（DEBT-002 + 联动 DEBT-006 AddressSpace refcount）。

证据：sys_exit.cpp:34-63（无 delete）/ process_new.cpp:153-168（reap 无 delete）/ scheduler.cpp:210-240（exit_current 无 delete）/ task_builder.cpp:153-170（release_resources 仅 operator delete）/ remove_task 仅 test（test_*.cpp）。

### DEBT-018（新）— D13 I2 fail：kMaxCpus 不一致

`kMaxCpus` 两处定义不同值不同类型：acpi.hpp `size_t=16`、percpu.hpp `uint32_t=8`。数组用不同值（percpu 系列 8 / cpu_apic_ids 16）。ap_main `cpu < proc::kMaxCpus`（8）保护避免 OOB，但：(1) ACPI 报 >8 CPU 时 AP 静默不启动（丢弃）；(2) 同名常量两值违反单一定义（ODR 风险，编译期可静默选其一）；(3) 类型不一致（size_t vs uint32_t）。P2 一致性。修：统一单一 kMaxCpus（建议 percpu=8 为权威，acpi 改用之；或显式 ACPI_MAX_CPUS 区分）。

证据：acpi.hpp:150 / percpu.hpp:28 / scheduler.cpp:42 / percpu.cpp:16 / gdt.cpp:17 / ap_main.cpp:189,260。

# F3-M2 批1 — TLS（fs_base 段基址）

> 2026-06-18。F3-M2（线程支持）批1：为每线程独立 TLS 打地基——CpuContext 加 `fs_base` 字段、context_switch.S 存/恢复 MSR_FS_BASE、`cinux::arch::set/get_tls_base` wrmsr/rdmsr 助手。1 批，783→785（+2 测试），fresh 785/0。批4 clone(CLONE_SETTLS) 与未来 pthread 消费此基建。

## 背景

x86-64 用 FS 段基址（MSR_FS_BASE=`0xC0000100`）实现 TLS：每个线程的 `%fs` 指向自己的 TLS 块。要在 CinuxOS 支持多线程（clone），必须让每个 Task 保有独立的 fs_base，并在上下文切换时存/恢复——否则线程 A 的 TLS 指针会随切换泄漏给线程 B。现有 CpuContext 已存 `gs_base`/`kgs_base`（swapgs 的 per-CPU 内核 GS 对，context_switch.S 已存恢复 0xC0000101/0102），但**无 fs_base**。

## 设计

- **CpuContext 加 `fs_base`（offset 80）**。CpuContext 是 `alignas(16)`，显式字段原 80B → 加 fs_base 成 88B → 对齐填充到 **96B**（`static_assert(sizeof==96)`；bytes 88..95 是未用对齐填充，asm 不碰）。同步更新 [test_scheduler.cpp](kernel/test/test_scheduler.cpp) 的布局断言（80→96 + 补 gs/kgs/fs 偏移）。
- **context_switch.S** 存/恢复 fs_base（MSR `0xC0000100`），紧跟现有 gs_base/kgs_base（`0xC0000101`/`0x0102`）同款 `rdmsr`/`wrmsr` 三段式（eax 低 32 / edx 高 32 / rcx=MSR 号）。
- **`kernel/arch/x86_64/tls.{hpp,cpp}`**：`set_tls_base(addr)`（wrmsr）/`get_tls_base()`（rdmsr），命名空间 `cinux::arch`，常量 `kMsrFsBase`。clone(CLONE_SETTLS) 调 `set_tls_base`。
- **task_builder** 新核线程显式 `ctx.fs_base=0`（无 TLS 直到 SETTLS）。**fork 不动 fs_base**——`memcpy` 继承父值（fork 子进程的 TLS 区被 CoW 拷贝，继承 fs_base 指向该拷贝，语义正确；区别于 gs_base=0/kgs_base=per_cpu 那种 kernel swapgs 状态不可继承）。

## 关键陷阱：wrmsr FS_BASE 须规范地址，否则 #GP（GOTCHA#17）

**首版测试用 `set_tls_base(0x1234567890ABCDEF)`，QEMU 立刻 #GP panic（err=0）挂在 wrmsr 上。** 寄存器现场：`RCX=0xC0000100`（MSR 号对）、`RDX=0x12345678`（高 32）、`RAX=0x90ABCDEF`（低 32）——操作数全对，是 wrmsr 指令本身抛 #GP。

**根因**：x86-64 要求 FS_BASE/GS_BASE 持有**规范地址（canonical）**——bits 48..63 必须符号扩展 bit 47（低半 `0x0000...` / 高半 `0xFFFF...`）。`0x1234567890ABCDEF` 的 bit 47=0 但 bit 48..63=`0x1234`（非全 0）→ 非规范 → wrmsr `#GP` err=0。

**诊断关键**：现场 RAX/RDX/RCX 全是 wrmsr 操作数且未进入下一条指令（RDI 仍持 `addr` 形参 `0x1234567890ABCDEF`）→ 故障点钉死在 wrmsr；排除操作数错误后只剩「MSR 不接受该值」→ 规范地址约束。

**修复**：测试改用规范地址（`0x0000123456789ABC` / `0x00007F0000000000`，bit 47=0 且高 16 位全 0）。真实 TLS 基址是用户指针，天然规范，clone(CLONE_SETTLS) 不受影响——**纯测试值错误，非产品 bug**。`tls.hpp` 文档注明此约束。

**此坑正是 timeout 40 的价值**：panic handler 死循环输出会挂死终端不返回，无 timeout 不会退出；`timeout 40` 杀进程、暴露非零退出码 + 现场寄存器，才快速定位（→ 触发用户要求把 timeout 40 写进验证规范，见 DIRECTIVES L5）。

## 验证

- 单测 +2：`test_set_get_roundtrip`（规范值 round-trip，验 eax/edx 两半）+ `test_high_bits_preserved`（不同 edx 值，验 rdmsr 返完整 64 位）。
- 全量：fresh `run-kernel-test`（`timeout 40` 包裹）783→**785/0**。全启动路径（init/gui/AHCI/ext2/shell）含 fs_base 上下文切换全程不炸——context_switch.S 的 fs_base 存恢复正确。

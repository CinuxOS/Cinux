# F10-M1 批5 — musl 源码编译 + 自包含 sysroot + 静态 hello

> 里程碑：F10-M1 用户态运行时 / musl 静态移植（批5）。分支 `feat/f10-musl`，commit `ea20c27`。
> 验证：端到端从零可复现 `build-musl.sh` → `build-hello.sh` → `./build/musl/hello` 输出
> `Hello from musl on CinuxOS!` exit 0；ELF ET_EXEC @0x400000 4 个 PT_LOAD。

## 背景

批1-4 把 syscall ABI / 结构体布局 / execve 初始栈 / musl 所需 syscall 全对齐了。批5 是
PLAN 标的「最大外部不确定项」：自己源码编译 musl（自包含 sysroot），用它 `-static` 编
hello world。纯 host 侧活，不改内核。批6 才把它丢进 QEMU 跑。

## 关键确认（先读码再动手）

- **CinuxOS 走真 ELF 加载链**：`launch_user_program(path)` → `execve(path)` 从 VFS/ext2 读真 ELF、
  解析 PT_LOAD、批3 铺初始栈、跳 entry（非-GUI 走 `/bin/sh`，见 `shell_launch.cpp`）。所以 musl
  静态 ELF（ET_EXEC + PT_LOAD）能走这条路——批6 验证。现有 `user_shell` 是另一条**内核内嵌
  flat blob** 机制（objcopy -O binary → ld -r 嵌镜像），跟 execve/ext2 加载是两回事。
- **host 工具链**：GCC 16.1.1 + GNU ld 2.46（比 handoff 记的 GCC14 新，能编 musl）。

## 三大坑（都踩了，写进 README）

1. **别给 musl configure 传 `--target`**。`./configure --target=x86_64` 会让 musl 给 binutils 加
   `x86_64-` 前缀（交叉模式），host 没有 `x86_64-ar` → 归档阶段 `make: x86_64-ar: No such file`
   （所有 `.o` 已编过，死在 `ar`）。正解：原生编译，显式 `CC=gcc AR=ar RANLIB=ranlib`，不传
   `--target`。
2. **GCC ≥ 14 别用 `musl-gcc` wrapper**。host specs 注入 `-latomic_asneeded`（GCC14+ 自带 spec），
   musl 的 `musl-gcc.specs` 没压住它 → `/usr/bin/ld: cannot find -latomic_asneeded`。正解：手动
   `-nostdlib` 链，绕开 wrapper。
3. **不能漏 `crtbeginS.o`/`crtendS.o`**。musl-gcc.specs 的 startfile/endfile 是
   `Scrt1.o crti.o crtbeginS.o … crtendS.o crtn.o`。手动链漏了 GCC 自己的 crtbeginS/crtendS
   （管 `.init_array`/`.fini_array`/`.eh_frame`）→ ELF 编出来结构完美（ET_EXEC + 正确 _start 反汇编
   + 全符号），但**启动期 segfault**（host 跑 exit 139）。补上 crtbeginS/crtendS（`gcc -print-file-name=`）
   立刻正常。这是个隐蔽坑：ELF 看着对、_start 对、符号全在，就是崩。

## 改动（tools/musl/，全 host 侧）

- **`build-musl.sh`**：下 `musl-1.2.5.tar.gz`（官方 tarball；cgit clone 会重定向）→ 解压 → 原生
  configure（坑1）→ `make -j` + `make install` → `build/musl-sysroot/`（libc.a 2.7M + crt1/Scrt1/
  rcrt1/crti/crtn.o + include/ + musl-gcc）。幂等（libc.a 在即跳过）。
- **`build-hello.sh`**：手动 `-nostdlib -static` 链（坑2/3），crt 顺序严格按 musl-gcc.specs：
  `Scrt1.o crti.o crtbeginS.o hello.c -lc -lgcc crtendS.o crtn.o`。产物 `build/musl/hello`。
- **`hello.c`** + **`README.md`**（三大坑 + 用法 + 批6 衔接说明）。

## 验证（端到端可复现，不依赖 /tmp 残留）

```sh
tools/musl/build-musl.sh   # 下载 + 编 musl → build/musl-sysroot
tools/musl/build-hello.sh  # 编 hello → build/musl/hello
./build/musl/hello         # -> Hello from musl on CinuxOS!  (exit 0)
```

ELF 结构（CinuxOS loader 视角）：ET_EXEC，entry `0x40103b`，4 个 PT_LOAD @ `0x400000`/`0x401000`/
`0x403000`/`0x404fc0`（R / R+E / R / RW），GNU_STACK RW 不执行。基址 `0x400000` 与 CinuxOS
约定一致（F9 note）。22KB（未 strip）。

## follow-up（批6 的事）

- 把 `build/musl/hello` 塞进 ext2 镜像（现 `/bin/sh` 在 ext2 上；hello 同理，路径如 `/hello`）。
- `execve("/hello")` → ELF loader 映 4 段 → 批3 初始栈（auxv）→ 跳 `_start`。
- musl `__init_tls`(arch_prctl ARCH_SET_FS) → `printf`(__stdio_write → writev) → exit_group，
  全走批4 syscall。QEMU 串口看到 `Hello from musl on CinuxOS!` 即端到端通。
- 这是批3（launch 铺栈）+ 批4（syscall 闭环）的真验证网——前三批的 in-kernel/host 单测只覆盖了
  能测的部分，真 musl 程序跑通才是闭环。

## sysroot 产物说明

`build/musl-sysroot/` + `build/musl/`（tarball/src/hello）全在 `build/`（gitignore `build*/` +
`*.o/*.a/*.elf`），**不入 git**。可复现：任何会话 `tools/musl/build-musl.sh && build-hello.sh` 即重建。

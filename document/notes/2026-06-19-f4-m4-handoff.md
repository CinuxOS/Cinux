## ⚠️ M4-1 试做关键发现(已回退,别重踩)
M4-1(current_→percpu)**不是「机械低风险」** —— 试做后 production 正常(run-smp 过)但**测试内核 hang**(卡 Sync Tests)。

**根因**:phantom-task 测试范式**靠 `current_`(静态)与 `percpu()->current` 的分歧**:
- 测试直接写 `percpu()->current = task`(**不**经 `set_current`,故不更新静态 `current_`);
- 旧 `Scheduler::block()` 用 `if (task == current_)` → `current_` 没被测试更新 → 判 **false** → **不 schedule**(测试期望:block 标 task 为 Blocked 即返回,无真实调度,见 test_futex.cpp:6 注释);
- M4-1 统一成 percpu 后,`block()` 读 `percpu()->current`(= task)→ 判 **true** → `schedule()` → runq 空 + idle_task_ 未被测试设 → **`while(1) cli;hlt` 死循环挂死**(scheduler.cpp schedule() 的 halt 分支)。

**涉及测试**:test_sync(竞争测试)、test_futex、test_clone(test_exit_cleartid)、test_sync_concurrent —— 都用 `percpu()->current = task` + `block(task)` 范式。这是 GOTCHA#21 家族的深层。

**结论**:**M4(含 M4-1)必须先重构 phantom-task 测试**,否则 current_→percpu 必挂测试。M4-1 代码本身正确(production 验证过),改法见下。

### M4-0 phantom-task 测试重构思路
当前测试:`percpu()->current = task`(直接);`Mutex::lock`/`futex_wait` 调 `block(task)` 期望不 schedule。重构方向(选一,需设计):
- (a) 测试改为**真实调度**:Scheduler::init 已建 idle_task_;让 block(current) schedule 到 idle(需测试在 idle 栈上观察被 block 的 task 状态)。改动大,断言要重写。
- (b) **block() 加「不切换」测试模式 / 标志**(hacky,不推荐)。
- (c) 测试用 **set_current + 一个 ready 的 next task**(让 schedule 有处可切),断言在 next 上做。
- 建议先读 test_sync.cpp / test_futex.cpp 全文 + Scheduler::block/schedule,定方案再改。**这是 M4 的真正前置,别跳过。**

## GOTCHA / 关键点
- **run-kernel-test ~60-90s**,timeout 用 120+(40s 会截尾,显示假象数字非回归)。
- **swapgs 纪律**(Phase 1):内核态 GS_BASE=&percpu_blocks[cpu]。M4 若让 AP `sti;hlt` 接 IPI,AP 的中断路径走 ISR 条件 swapgs(已就绪)。
- **AP cli;hlt vs sti;hlt**:M4-2 改 sti;hlt 后,AP 会收中断(PIT 等),中断 handler 若碰共享 `Scheduler::current_`(M4-1 前)或 runq → 并发。故 **M4-1(current_ per-CPU)必先于 M4-2**。
- lost-wakeup 窗口(M4-3):`读条件→enqueue→block()` 之间另一核 unblock 会被 block() 的 state=Blocked 覆盖。
- QEMU:-smp 2 用 `run-smp`(qemu.cmake target);单核 `run`(boot_aps no-op)。

## 目标

buddy 之上分层小对象分配器，**全替 Heap**（不留 fallback），闭环
`PMM(buddy) → Slab → kmalloc/kfree → operator new`。

- 小对象（effective `max(size, align) ≤ 2048`）→ Slab 通用缓存（8 档 16-2048B）。
- 大对象 / 大对齐 → buddy `alloc_pages` + **direct-map 复用**（`virt = phys + DIRECT_MAP_BASE`，零 map / 零元数据，DmaPool 同款 GOTCHA#7/#13）。

## 关键设计点

- **侵入式 freelist 写 4K 页**：slab 页走 `KMEM_SLAB`（PML4[256]，4K 映射），**绝不 direct-map huge**（PML4[272]，sub-page 写读在 nested-KVM 不一致，GOTCHA#13/#14）。`grow_cache` 映页后整页 `memzero`（干净毒检基线 + 无 stale 泄漏）。
- **double-free 毒检**（O(1)）：freed 对象 word[1] 写 sentinel；fresh/allocated 对象被 zero，故命中即重复 free。
- **类专属 operator new/delete**：Task/VMA/CachedPage 各加 4 重载（scalar + aligned），所有 `new T`/`delete p` 自动路由到各自专用缓存，**无调用点改动**（错误路径自动覆盖）。
- **专用缓存支持非幂 obj_size**：`kObjAlign=16` 统一对象对齐，header 区也按 16 对齐 → 任意 obj_size（如 Task 1008B、VMA 56B）都能密集排布。实测 Task 1008B→4/slab（原 1024 档 3）、VMA 56B→72/slab（原 64 档 63），真省碎片。
- **IF=0 安全**：`alloc/free` 用 `irq_guard`（IF=0 幂等），PF handler 的 `new CachedPage` 走 slab 安全（同 Heap::expand 在 IF=0 的既有路径）。

## Inode 不入 slab

ext2 用固定 `inode_cache_[EXT2_INODE_CACHE_SIZE]` 数组自管 VFS inode（`kernel/fs/ext2.hpp:399`）——非堆分配，故无 slab 专用缓存（N/A，非遗漏）。这与 slab 是正交的两套缓存机制。

## ⚠️ 修了一个真 bug：page_cache 陈旧命中（GOTCHA#15）

批2 `test_shell_write` 红：写 `"hello world\n"` 读回 `"Hello from e"`。

- **根因非 slab**（slab 单测全过）：page cache 原按 `Inode*` 指针做 hash/lookup 键。slab 正确复用已释放 Inode 的内存地址给新 Inode → 新文件查 cache 时命中**陈旧页**（旧文件内容）→ `sys_read` 返错字节。Heap first-fit 侥幸不复用同地址，故潜伏至今。
- **修**：page_cache 改按 `inode->ino`（稳定号）键控（`hash_key`/`lookup_locked`/`insert`）。
- **通用铁律**：任何按对象指针/地址键控的在线结构（cache/table），对象经分配器回收时必然陈旧命中，键须用稳定 id/number 而非 pointer。

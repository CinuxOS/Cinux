`default_rr_` 是全局单例，所有 CPU 共用 + `RoundRobin::lock_`(Spinlock+irq_guard) 跨核互斥**已就绪**。M4-2 **不新建 runq**。三件事：
1. **reschedule IPI**：把 idle AP 从 `hlt` 踢醒。
2. **唤醒点**：`add_task`/`unblock` 后发 IPI 给 idle AP。
3. **AP idle loop**：AP `sti;hlt`，hlt 醒来后 `schedule()` 取共享 runq（有则切去跑、无则再 hlt）。

## M4-2-2：AP idle loop + 首次切换 + -smp 2 真机（高风险）

### 改动
1. **AP idle loop**（[ap_main.cpp:136](kernel/arch/x86_64/ap_main.cpp#L136) 替换 `cli;hlt` 死循环）。骨架：
   ```c
   percpu()->current = <AP idle task>;   // 见下：idle task 决策
   while (true) {
       Scheduler::schedule();            // 共享 runq 有任务则切去跑；无则返回继续
       __asm__ volatile("sti; hlt");     // 睡到下一个中断（reschedule IPI）
       __asm__ volatile("cli");
   }
   ```
2. **首次切换的 prev（关键设计点）**：AP 第一次 `schedule()` 切到真实任务时，没有前驱 task 上下文。需要 **AP idle task 作 prev**（`context_switch(&idle->ctx, &real->ctx)`）。决策：复用全局 `idle_task_`（其 `idle_entry` 是 `while(hlt)`，无 per-CPU 状态，共享 OK）还是 per-CPU 建。**倾向**：AP 设 `percpu()->current = idle_task_`（共享，entry 纯 hlt 无状态），schedule 从 runq 取真实任务切走。**GOTCHA#23**：context_switch 恢复点读 `current()`（per-CPU），勿用局部 next/prev。
3. **sti;hlt 原子性**：x86 `sti` 后紧跟 `hlt`，sti-hlt 间到达的中断在 hlt 前服务（hlt 不睡）。但 idle loop 应在 `hlt` 前重检 runq，防「设 idle 后、hlt 前有任务入队但 IPI 漏发」窗口。
4. **idle 跟踪并发**：若用精确 idle 标志（percpu[].idle），BSP `wake_idle_ap` 读、AP idle loop 写 → 竞态。**倾向多发 IPI 简化**（见 M4-2-1.6），免精确跟踪。

## GOTCHA（M4-2 相关）
- **#23（既有）**：context_switch 恢复点 current 读 per-CPU（`current()`），勿用局部 next/prev。AP idle loop 首次切换 + 所有 schedule 切换都适用。
- **#24（既有，M4-3）**：prepare-to-wait 的 `schedule_blocked` 须在 irq_guard 析构后。M4-2 不直接碰，但 AP 真跑任务后会触发并发（M4-3 修复的验证对象）。
- **sti;hlt 原子性**：x86 保证 sti-hlt 间中断不漏，但 idle loop 须 hlt 前检查 runq。
- **idle 跟踪并发**：倾向多发 IPI（简单、无害）规避精确跟踪竞态。
- **APIC timer 未做**：AP 任务不被 tick 抢占（cooperative），CPU-bound 任务独占 AP 直到 yield/block/exit。**已定不做**（follow-up）。

## 不做（follow-up，已定）
- per-CPU APIC timer 时间片抢占（用户决策，PLAN 标 follow-up）。
- waitpid 彻底 SMP 安全（children 链表锁 + double-check；M4-3 只缩小窗口，waitpid 注释已记）。
- per-CPU runq + work stealing：M4-2 用全局共享 runq（简单正确）；per-CPU 留性能优化。

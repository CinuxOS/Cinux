# F6-M2 ProcFS — /proc 进程自省虚拟文件系统

> 里程碑:F6 VFS 第二里程碑。2026-06-30 收官。分支 `worktree-f6-m2-procfs`(从干净 main `c0188cd`)。
> 范围:把内核 Task registry 暴露成 `/proc`,提供进程自省。照抄 F6-M3 DevFs 范式。

## 背景 / 目标

F6-M3 DevFs(已合 main PR#48)建立了「内存型虚拟 FS」范式:`FileSystem` 子类 + 匿名 namespace `InodeOps` 子类 + boot 接线单独 `_init.cpp`。F6-M2 把同一范式用到进程自省:`/proc` 根目录枚举活进程 pid,`/proc/<pid>/` 暴露每进程伪文件。

**范围栅栏(不投机)**:本里程碑只做进程自省第一刀——`/proc` 根 readdir 枚举 pid + `/proc/<pid>/` 目录 + `/proc/<pid>/{stat,cmdline}` 动态伪文件。**不做**静态信息节点(`/proc/version`/`meminfo`/`cpuinfo`/`uptime`)、不做 `/proc/<pid>/{maps,fd,status}`、不做完整 Linux `/proc/<pid>/stat` 52 字段(CinuxOS 无 per-task accounting)——留 follow-up(见 `document/todo/f6-vfs/01-procfs.md`)。

## 设计

### ProcFs(FileSystem 子类)+ 定长 pid 索引 inode 池

DevFs 根目录是定长节点表;ProcFs 根目录条目是**动态 pid**(Task registry 实时快照)。关键差异在 inode 身份:

- **close 只删 File 不删 Inode**([file.cpp](../../kernel/fs/file.cpp) `delete fds_[fd]`)→ 每个 lookup 返回的 inode 必须由 ProcFs 拥有并长存。
- **PID_MAX = 256** 有界([pid.hpp](../../kernel/proc/pid.hpp))→ ProcFs 持 `pid_dir_inodes_[257]` / `stat_inodes_[257]` / `cmdline_inodes_[257]` 定长池,pid 索引,`ino = pid`、`fs_private = this`。
- **每 pid 一个稳定 inode** → 并发 lookup 不同 pid 不竞态(SMP 安全);无单 scratch 复用导致的 pid 覆写错读。
- `static_assert(kProcPidMax == PidAllocator::PID_MAX)` 锁定池上界,allocator 改了立刻编译期报。

### lookup(strip `/` 后解析)

`vfs_resolve("/proc/...")` 把 `/proc` 前缀剥掉,rel_path 形如 `""` / `"/1"` / `"/1/stat"`。ProcFs::lookup:
1. strip 单个前导 `/`;
2. `parse_pid` 解析前导十进制(溢出 PID_MAX 即拒,永不越界索引池);
3. `<pid>` → pid 目录 inode;`<pid>/stat` / `<pid>/cmdline` → 伪文件 inode;
4. **每条路径都经 `signal_find_task_by_pid` 校验存活**——非活 pid → NotFound(对齐 Linux,只露活进程)。

### readdir:活 pid 枚举

新增 accessor **`signal_nth_task_pid(n, *pid)`**(signal.hpp/cpp,**纯增量**,`g_registry_lock` 下走到第 n 个 task,不改 register/unregister/find_by_pid)。`ProcRootDirOps::readdir` 索引 0/1 = `.` / `..`,索引 i≥2 = 第 (i-2) 个活 pid(`utoa` 写十进制)。

### 伪文件内容(procfs_content.{hpp,cpp})

- `/proc/<pid>/stat` = `pid (name) state ppid tgid uid gid\n`(Linux 简化子集;state 字符 R/S/T/Z/X)。
- `/proc/<pid>/cmdline` = task name + NUL(CinuxOS Task 不存 argv,best-effort 暴露 comm)。
- read 时 `signal_find_task_by_pid` → Task,生成内容到栈 buffer,`copy_pseudo` 按 offset 拷贝(支持 pread / EOF)。

## 关键决策 / 陷阱

### 1. snapshot accessor → nth accessor(frame-size)

初版用 `signal_enumerate_task_pids(int* out, max)` 快照,readdir 在栈上 `int pids[kProcPidMax+1]` = 1028B,**超 kernel `-Wframe-larger-than=1024`**(production big_kernel_common 是 `-Werror`,不像 test/ 是 warning)。改 `signal_nth_task_pid`:锁内走到第 n 个,无栈数组,frame 小。代价:全 /proc 列举 O(tasks²),但 tasks ≤ 256 可忽略。**正解不踩 frame 限。**

### 2. 测试内核主线程不经 add_task → registry 可能空

[main_test.cpp](../../kernel/test/main_test.cpp) 在 boot 栈直跑,不进 registry;早期 scheduler 测可能留残留 task。`run_procfs_tests` 的 readdir/lookup 测**自建栈 Task 注册已知 pid**(`pick_free_pid` 在 [200,256] 找空位),不依赖 registry 残留状态,确定性验证(参 F3-M4 GOTCHA#22 栈 Task 思路)。

### 3. registry TOCTOU(已知,留 follow-up)

`signal_find_task_by_pid` 返指针后 unlock,read Task 字段时 task 可能已 exit+free(DEBT-002 RCU-safe registry 未修)。窗口极小(find 后立即快照字段),hobby OS 可接受;`test_stat_read_dead_pid_is_not_found` 实证:unregister 后读同一 inode → NotFound。真修随 DEBT-002。

### 4. 拆文件(500 行软限)

procfs.cpp 批3 涨到 504 行(CODING-TASTE §7b 红线)。按职责拆出 [procfs_content.{hpp,cpp}](../../kernel/fs/procfs_content.hpp)(Task→文本生成器,107+47 行),procfs.cpp 留 FS plumbing(inode 池/ops/lookup,406 行)。

### 5. §14 文件门

procfs.cpp 读 kernel registry(signal.hpp/process.hpp),**不能 host 单测**(DevFs 靠注入 CharSink 才 host-testable;ProcFs 直读 registry 无注入缝)。走 kernel harness(QEMU `run_procfs_tests`,11 测)。boot 接线(kprintf)独立 procfs_init.cpp,CMake 决定编不编,源码零 `#ifdef`。

## 验证

- **kernel 测**:`timeout 120 cmake --build build --target run-kernel-test-all -j$(nproc)` 两 leg **997/0**(基线 986 + 11 procfs)。本地 `cmake -B build -DCINUX_MUSL_HELLO_SMOKE=OFF -DCINUX_MUSL_DYN_SMOKE=OFF` 关 smoke 防挂死。
- **生产 boot 冒烟**(headless QEMU,tcg,timeout 40):
  ```
  [VFS] ext2 mounted at /
  [DEVFS] mounted at /dev (4 nodes)
  [PROCFS] mounted at /proc
  ```
  零 panic。procfs::init() 在 devfs::init() 后挂载。
- **全量编绿**(改公共头 signal.hpp,push 前补全量含 host)。

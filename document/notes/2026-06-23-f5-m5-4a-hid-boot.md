# 2026-06-23 F5-M5 批4A — HID boot mouse(Configure Endpoint + interrupt-in 轮询)【EP 配置成功】

## 背景 / 目标

3C 让设备能控制传输(读描述符/配置)。4A 把 HID boot 鼠标的 interrupt-in 端点配起来:读 config 描述符找鼠标的 interrupt-in EP → SET_PROTOCOL(boot)→ **Configure Endpoint 命令**(新,加 interrupt EP)→ interrupt-in 轮询 + 报告解码。这是「能用 USB 鼠标」的协议层最后一块(注入队列 + make run 留 4B/5A)。

## 验证 workflow(ultracode:多源 + 对抗)

8-agent workflow(QEMU hcd-xhci.c + Linux xhci-ring.c/xhci.h + USB HID spec)核实 4A,**3/3 对抗 lens 确认 Configure Endpoint input-context 构造**。关键坐实:
- **QEMU 不自动配置非 EP0 端点——Configure Endpoint(TRB type 12)必需**(非 no-op)。
- **加 EP1-IN 的 input context**:ICC drop=0、**add=A0|A3(=0x9)**(A1 必须清——EP0 已存在;QEMU 硬门 `(ictl[1]&0x3)!=0x1 → CC_TRB_ERROR`)。slot last_ctx=3(新 EP 的 DCI)。EP1-IN context 在 in[32..36](offset 0x80 = ictx+32+32*3)。
- **interval = 描述符 bInterval 直接传**(FS 规则,非 HS 2^ 转换)。EP_STATE 写 0(disabled),HC 自己翻 RUNNING。
- **Normal TRB(type1)轮询**,无 DIR 位(方向来自 EP context type),IOC+ISP。doorbell=DCI=ep*2+1(EP1-IN=3)。
- **usb-mouse 静止 NAK**(GOTCHA #7):无 Transfer Event 直到移动——测试不能强求报告。

## 关键陷阱(GOTCHA)

- **usb-mouse 静止 NAK**(GOTCHA #7):无 Transfer Event 直到 `usb_wakeup`(移动)。`poll_mouse_report` 返回 TimedOut 是**正确**的中断行为,非 bug。真报告证明留 5A(make run 移鼠标)。
- **input context buffer 太小会 CC_TRB_ERROR**:3B 的 96B 只到 dword 23;EP1-IN context 在 dword 32 → 扩到 160B。device context 同理扩到 128B(slot+EP0+EP1out+EP1in)。
- **interval 速度相关**:FS 鼠标直接传 bInterval(QEMU 实测=7,非 workflow 猜的 0x0A——我用描述符原值,对)。
- **host 测 `1<<10` 又算错**:Normal type=1 → 1<<10=0x400(非 0x800,那是 type=2)。interrupt TRB = 0x424 非 0x824。builder 对,测试期望值错(同 3C 的 hex 笔误教训)。
